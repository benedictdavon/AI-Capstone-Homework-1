# Personal Finance Expense Prediction ML

A compact machine learning mini-project for predicting daily personal expenses and discovering spending patterns from synthetic student finance data. The project uses a generated one-year transaction dataset, supervised regression for expense forecasting, and clustering methods for spending-profile analysis.

This repository was created for **AI Capstone Homework 1** and has been organized as a reproducible portfolio project rather than only a class report.

## Results Snapshot

## Problem Statement

Personal finance data is often noisy, seasonal, and highly dependent on routines. This project asks two related questions:

1. Can recent spending history and calendar features predict daily expense totals?
2. Can clustering reveal interpretable spending patterns such as low-spend days, routine days, and high-expense outliers?

The project is intentionally scoped as a small ML workflow: data generation, exploration, feature engineering, model evaluation, clustering, and result visualization.

## Dataset

The dataset is synthetic and generated by `generate_data.py` with `random.seed(42)`. It simulates one student's transactions from **2025-01-01** through **2025-12-31**.

Committed dataset:

- `data/daily_expenses.csv`
- 1,061 transaction rows
- Columns: `Date`, `Category`, `Amount_NTD`, `Description`, `Payment_Method`, `Time`
- Currency: New Taiwan Dollars (NTD)

Generated categories include:

- Income: scholarship, parental support, part-time work
- Fixed or recurring expenses: transport, entertainment subscriptions, groceries
- Variable daily expenses: lunch, dinner, coffee, laundry
- Occasional expenses: gym supplements and special food events

Important limitation: this is not real personal banking data. It is a controlled synthetic dataset intended for modeling practice and reproducibility.

## Methods

### Supervised Learning

The supervised task predicts daily expense totals using a Random Forest regressor from scikit-learn.

Feature engineering includes:

- `DayOfWeek`
- `IsWeekend`
- `Month`
- `Day`
- `Lag1`, previous-day expense
- `Rolling7`, 7-day rolling average
- log-transformed expense values for model training and evaluation

Model selection uses:

- `GridSearchCV`
- `TimeSeriesSplit`
- RMSE as the primary evaluation metric
- feature importance analysis for interpretability

### Unsupervised Learning

The unsupervised workflow explores spending behavior using:

- K-Means clustering
- Gaussian Mixture Models (GMM)
- PCA for two-dimensional visualization
- Elbow method, Silhouette score, Davies-Bouldin Index, and Calinski-Harabasz Index

The notebook reports an elbow-method choice of **k = 6**, while other clustering metrics prefer different values. That disagreement is useful context: the synthetic data has structure, but cluster count selection is not unambiguous.

## Run Commands

The repository already includes `data/daily_expenses.csv`, so the training notebook can be opened directly after setup.

Generate a fresh synthetic dataset:

```bash
python generate_data.py
```

Note: `generate_data.py` currently writes `daily_expenses.csv` to the repository root. The training notebook expects `data/daily_expenses.csv`, while the ETL notebook reads `daily_expenses.csv` from the root. To use a regenerated dataset for training, copy or move it into `data/`:

```bash
# macOS/Linux
mkdir -p data
cp daily_expenses.csv data/daily_expenses.csv

# Windows PowerShell
New-Item -ItemType Directory -Force data
Copy-Item daily_expenses.csv data\daily_expenses.csv -Force
```

Open the notebooks:

```bash
jupyter lab "ETL & Exploration.ipynb"
jupyter lab training.ipynb
```

Expected notebook workflow:

1. Run `ETL & Exploration.ipynb` for initial inspection of `daily_expenses.csv`.
2. Run `training.ipynb` for feature engineering, Random Forest experiments, K-Means, GMM, PCA visualizations, and result export.
3. Review generated outputs in `data/experiment_result/`.

Manual verification note: the current project is notebook-first. The commands above describe the expected usage, but this cleanup did not convert notebook cells into a single command-line training script.

## Limitations

- The dataset is synthetic and rule-based, so model performance does not prove real-world personal finance accuracy.
- The spending behavior represents one simulated student profile, not a broad population.
- Some feature engineering choices, such as lag and rolling averages, depend on clean daily aggregation.
- Clustering metrics disagree on the best number of clusters, so cluster interpretation should be treated as exploratory.
- The workflow is notebook-based rather than packaged as a fully automated pipeline.
- Data path conventions are not fully standardized between the generator, ETL notebook, and training notebook.
